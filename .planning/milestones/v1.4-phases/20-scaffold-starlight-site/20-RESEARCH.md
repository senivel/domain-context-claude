## Summary

Starlight is Astro's official documentation framework. The current versions are Astro 6.0.5 and @astrojs/starlight 0.38.1 (published 2026-03-11). Starlight provides all required features out of the box with zero custom code: sidebar navigation, Pagefind full-text search, dark/light mode toggle, Expressive Code syntax highlighting with copy button, and responsive layout.

The key architectural decision is placing the entire Astro project inside a `docs/` subdirectory with its own `package.json`, completely isolated from the root npm package. The root `package.json` already uses a `files` whitelist that excludes `docs/`, so npm pack exclusion is automatic. The `.gitignore` needs entries for Astro build artifacts inside `docs/`.

**Primary recommendation:** Use `npm create astro@latest -- --template starlight` inside a `docs/` directory, configure `astro.config.mjs` with the project title and GitHub social link, and add Astro-specific entries to `.gitignore`.

## Architecture Patterns

### Recommended Project Structure (docs/ subdirectory)
```
docs/
├── astro.config.mjs     # Starlight integration config
├── package.json          # Isolated dependencies (astro + starlight)
├── tsconfig.json         # TypeScript config (generated by template)
├── src/
│   ├── assets/           # Images, logos
│   ├── content/
│   │   └── docs/
│   │       └── index.mdx # Homepage / placeholder page
│   └── content.config.ts # Content collection schema
└── public/               # Static assets (favicon, etc.)
```

### Pattern 1: Isolated Subdirectory Project
**What:** The `docs/` directory is a fully independent Astro project with its own `package.json`, `node_modules/`, and build tooling. It shares nothing with the root project.
**When to use:** When the main project is an npm package and docs must not affect it.
**Key config -- `docs/astro.config.mjs`:**
```javascript
import { defineConfig } from 'astro/config';
import starlight from '@astrojs/starlight';

export default defineConfig({
  integrations: [
    starlight({
      title: 'Domain Context',
      social: {
        github: 'https://github.com/senivel/domain-context-claude',
      },
      sidebar: [
        {
          label: 'Getting Started',
          autogenerate: { directory: 'getting-started' },
        },
      ],
    }),
  ],
});
```

### Pattern 2: Content Collection Schema
**What:** Starlight requires a `src/content.config.ts` file defining the docs collection using Starlight's loaders and schemas.
**Config -- `docs/src/content.config.ts`:**
```typescript
import { defineCollection } from 'astro:content';
import { docsLoader } from '@astrojs/starlight/loaders';
import { docsSchema } from '@astrojs/starlight/schema';

export const collections = {
  docs: defineCollection({
    loader: docsLoader(),
    schema: docsSchema(),
  }),
};
```

### Pattern 3: Placeholder Index Page
**What:** Minimal frontmatter-driven index page that proves the site works.
**File -- `docs/src/content/docs/index.mdx`:**
```markdown
---
title: Domain Context for Claude Code
description: Codify and maintain domain knowledge alongside code so AI assistants always have accurate business context.
template: splash
hero:
  tagline: Documentation site — coming soon
  actions:
    - text: GitHub
      link: https://github.com/senivel/domain-context-claude
      icon: external
---
```

### Anti-Patterns to Avoid
- **Sharing node_modules with root:** Never hoist docs dependencies to root. The docs/ project must be fully self-contained.
- **Adding docs scripts to root package.json:** Keep root package.json unchanged. Developers `cd docs && npm run dev`.
- **Custom CSS for framework defaults:** All five required features (sidebar, search, dark/light, code highlighting, responsive) work without any custom CSS. Adding custom styles risks breaking on upgrades.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Full-text search | Custom search indexer | Pagefind (Starlight default) | Pagefind indexes at build time, works offline, zero config |
| Dark/light mode | Theme toggle component | Starlight built-in toggle | Handles system preference detection, persistence, SSR |
| Code highlighting | Prism/Highlight.js setup | Expressive Code (Starlight default) | Copy button, line highlighting, file name tabs all included |
| Sidebar navigation | Manual nav component | Starlight sidebar config | Auto-generates from directory structure or manual config |
| Responsive layout | Media queries | Starlight default CSS | Mobile-first, accessible, tested across browsers |

**Key insight:** Every feature required by INFRA-04 is a Starlight default. Zero custom code needed.

## Common Pitfalls

### Pitfall 1: Node.js Version Incompatibility
**What goes wrong:** Astro 6 requires `^20.19.1 || >=22.12.0`. Older Node 20.x versions (< 20.19.1) will fail.
**Why it happens:** Astro 6 uses newer Node APIs.
**How to avoid:** Verify `node --version` satisfies the constraint before setup. Current environment is Node 24.14.0 (compatible).
**Warning signs:** Cryptic startup errors about unsupported features.

### Pitfall 2: Forgetting .gitignore Entries for Astro
**What goes wrong:** Astro build artifacts (`docs/dist/`, `docs/.astro/`) get committed.
**Why it happens:** Root `.gitignore` has no Astro-specific patterns.
**How to avoid:** Add `docs/dist/`, `docs/.astro/`, and `docs/node_modules/` to root `.gitignore`.
**Warning signs:** Large binary files showing up in git status.

### Pitfall 3: Running npm Commands from Wrong Directory
**What goes wrong:** Running `npm install` or `npm run dev` from root instead of `docs/`.
**Why it happens:** Developers forget the docs site is a separate project.
**How to avoid:** Document clearly that all docs commands run inside `docs/`.
**Warning signs:** "Missing script" errors, dependencies installed at root.

### Pitfall 4: Content Collection Config Missing or Wrong
**What goes wrong:** Pages don't render or throw build errors.
**Why it happens:** Starlight requires `src/content.config.ts` with specific loader/schema imports.
**How to avoid:** Use the exact content.config.ts pattern from Starlight docs (shown in Architecture Patterns above).
**Warning signs:** "Collection not found" or schema validation errors.

```bash

npm pack --dry-run 2>&1 | grep docs
# Expected: no output (docs/ is excluded by files whitelist)
```

```bash
cd docs && npm run dev
# Serves on http://localhost:4321 by default
```
