Create `commands/dc/refresh.md` following the established skill file format (YAML frontmatter + objective/execution_context/process sections).

**Frontmatter:**
- name: dc:refresh
- description: Review stale domain context entries and update them based on current code reality. Identifies entries older than 90 days, assesses accuracy against source code, and proposes updates.
- allowed-tools: Read, Write, Edit, Grep, Glob, AskUserQuestion

**Process steps (must implement ALL of these):**

**Step 1: Check for .context/ directory.** If missing, display "No .context/ directory found. Run /dc:init to set up domain context." and stop. Same pattern as dc:validate Step 1.

**Step 2: Read and parse MANIFEST.md.** Parse entries under all four section headers (Domain Concepts, Architecture Decisions, Constraints, Module Context Files). For each entry, extract: name, file path, verified date, section. Skip `(none yet)` placeholder lines. Same parsing logic as dc:validate Step 2.

**Step 3: Identify stale entries.** For each parsed entry:
- If verified date exists and is >90 days old: mark as stale, record days count.
- If no verified date: mark as stale (treat same as dc:validate).
- Track total entry count for the clean-state message.

**Step 4: Handle clean state.** If no stale entries found, display: "All entries are fresh. {N} entries checked, none older than 90 days." Then stop.

**Step 5: Process each stale entry one at a time.** For each stale entry, in order:

  **5a: Verify file exists.** Read the entry's file (resolve path relative to .context/). If file not found, display "File not found at {path} -- run /dc:validate to fix broken links first." and skip to the next entry.

  **5b: Read entry content.** Read the full content of the context file. Identify the key claims and structured sections.

  **5c: Find relevant source code.** Extract search terms from the entry's structured sections:
  - For domain concepts: Related Concepts, Key Attributes, Business Rules sections.
  - For decisions: Affected Modules, Decision sections.
  - For constraints: Impact on Code, Requirements sections.
  - Fallback: use the entry title as a grep term.
  Use Grep to search the codebase for 2-3 targeted terms. Exclude .context/, .planning/, node_modules/, .git/ directories. Show up to 3 most relevant snippets, each max 10-15 lines.

  **5d: Assess accuracy.** Compare entry content against found source code. Determine if the entry is:
  - "Still accurate" -- content describes current behavior correctly, no factual contradictions.
  - "Drifted" -- content contradicts current code or misses significant changes.
  Show brief reasoning (e.g., "Entry describes X, code still does X" or "Entry says X but code now does Y").

  **5e: Present to user.** Display:
  - Entry name, days stale, file path.
  - Key claims from the entry (2-3 bullet points summarizing what the entry says).
  - Relevant code snippets found.
  - Accuracy assessment with reasoning.

  **5f: User decision via AskUserQuestion.**
  - If still accurate: prompt with options "Still accurate -- bump date", "Edit first", "Skip this entry".
  - If drifted: show proposed changes in diff format (old lines prefixed with -, new lines prefixed with +), then prompt with options "Apply changes", "Edit first", "Skip this entry".

  **5g: Execute user's choice.**
  - "Bump date": Update verified date in BOTH MANIFEST.md (`[verified: old-date]` -> `[verified: today]`) AND the context file (`<!-- verified: old-date -->` -> `<!-- verified: today -->`). If the context file lacks the `<!-- verified: -->` comment, add it on line 3.
  - "Apply changes": Write the proposed content updates to the context file using Edit, then bump the verified date in both locations.
  - "Edit first": Use AskUserQuestion to ask "What would you like to change?" Collect freeform response, apply edits, then bump verified date in both locations.
  - "Skip": Move to next entry, no changes.

  Track all modified files for the commit step.

**Step 6: Summary.** After all entries are processed, if any changes were made, display:
```
Refresh complete:
  {N} entries reviewed
  {M} dates updated
  {K} entries updated with new content
  {J} entries skipped
```

**Step 7: Commit prompt.** Only if changes were made. Use AskUserQuestion: "Commit the changes?" with options "Yes (recommended)", "No". If "Yes": git add all modified files (context files + .context/MANIFEST.md), git commit with message `docs: refresh domain context -- {N} entries updated`.

**Important constraints (from user decisions):**
- Per-entry confirmation required -- no batch skip option.
- Minor wording improvements don't count as drift -- only factual contradictions.
- Show relevant code snippets alongside entry content -- enough for user to agree/disagree, not full files.
- Inline diff format for proposed changes: section header, old content (prefixed -), new content (prefixed +).
- For module context files, grep within the module's directory only (parent directory of the CONTEXT.md file).
