Create `commands/dc/add.md` as a Claude Code skill file following the exact format of dc:init and dc:validate. The skill implements this flow:

**Frontmatter:**
```yaml
---
name: dc:add
description: Add a new domain concept, architecture decision, or constraint from a freeform description. Creates the file and updates MANIFEST.md.
allowed-tools:
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - AskUserQuestion
---
```

**Objective section:** Brief description of what dc:add does.

**Execution context section:** Document:
- Template resolution (same as dc:init Step 1)
- Template files used: domain-concept.md, decision.md, constraint.md
- MANIFEST.md section headers and entry line formats (copy from dc:validate execution_context)
- Path-to-section mapping: domain/ -> Domain Concepts, decisions/ -> Architecture Decisions, constraints/ -> Constraints
- Access level behavior: public = .context/, private = .context.local/

**Process section with these steps:**

**Step 1: Check for .context/ directory.** If missing, suggest /dc:init. Stop.

**Step 2: Resolve template path.** Same logic as dc:init Step 1 (local then global). If neither exists, error with install instructions. Stop.

**Step 3: Determine entry type.** Check if type was provided as argument (skill invocation argument). Accept "concept" / "domain concept", "decision", or "constraint" (case-insensitive). If no argument or unrecognized, use AskUserQuestion:
- Prompt: "What type of entry would you like to add?"
- Options: "Domain concept", "Architecture decision", "Constraint"

**Step 4: Collect freeform description.** Use AskUserQuestion:
- Prompt: "Describe the [type] you want to add. Include as much detail as you'd like -- I'll organize it into the right template sections."
- Options: 1. (freeform text area -- recommended)

**Step 5: Extract title and content.** From the freeform description:
1. Extract or infer a concise title for the entry.
2. Read the appropriate template from TEMPLATE_DIR.
3. Map the user's description to template sections. For each section:
   - If the user mentioned relevant content, fill that section.
   - If the user did not mention it, fill with "Not yet documented."
4. Strip all HTML comments from the template content.
5. Replace `{verified_date}` with today's date (YYYY-MM-DD format).
6. For decisions: default `{status}` to "accepted" (per locked decision). For ADR number, see Step 6.
7. Auto-derive kebab-case filename from the title (per locked decision -- no user prompt for filename):
   - Lowercase, replace spaces with hyphens, remove special characters except hyphens, collapse multiple hyphens, trim leading/trailing hyphens.

**Step 6: ADR number auto-detection (decisions only).**
1. Use Glob to find `.context/decisions/[0-9][0-9][0-9]-*.md`
2. For each file, extract the first 3 characters as a number.
3. Next number = max of all numbers + 1. If no files found, start at 001.
4. Zero-pad to 3 digits.
5. Prepend to the kebab-case filename: `{NNN}-{kebab-title}.md`
6. Replace `{number}` in the template with the padded number.

**Step 7: Detect access level.**
1. Scan the user's freeform description for keywords: "confidential", "private", "secret", "internal-only", "restricted" (case-insensitive).
2. If found, use AskUserQuestion: "This sounds like it might be confidential. Should this be a private entry in .context.local/?" with options "Yes, make it private" and "No, keep it public".
3. If no keywords found, default to public.
4. Set target directory: public = `.context/{subdir}/`, private = `.context.local/{subdir}/`.

**Step 8: Duplicate detection.**
1. Construct the full file path: `{target_dir}/{filename}.md` (or `{NNN}-{filename}.md` for decisions).
2. Attempt to read the file. If it exists, warn user: "A file already exists at {path}. Aborting to avoid overwriting." Stop.
3. Read `.context/MANIFEST.md` (and `.context.local/MANIFEST.md` if private). Parse entries in the target section. Check if any entry name matches the extracted title (case-insensitive). If match found, warn: "An entry named '{name}' already exists in MANIFEST.md. Aborting." Stop.

**Step 9: Preview.**
1. Show the complete file content that will be written.
2. Show the MANIFEST.md entry line that will be added.
3. Show the file path where it will be created.
4. Use AskUserQuestion: "Create this entry?"
   - Options: "Accept (recommended)", "Edit"
5. If "Edit": use a new AskUserQuestion asking "What would you like to change? Describe your corrections." Collect freeform input, re-extract/adjust content, show preview again with "Accept" / "Cancel" options only (one round of editing per locked decision from research).
6. If "Cancel" at any point: stop without writing.

**Step 10: Write file.**
1. For private entries: run `mkdir -p` on the target directory (e.g., `.context.local/domain/`).
2. Write the filled template content to the target path.

**Step 11: Update MANIFEST.md.**
1. Determine which MANIFEST.md to update: `.context/MANIFEST.md` for public, `.context.local/MANIFEST.md` for private.
2. For private: if `.context.local/MANIFEST.md` does not exist, create it with the basic section structure (copy from manifest template, fill description with "Private domain context entries", strip HTML comments).
3. Read the target MANIFEST.md.
4. Find the correct section header (## Domain Concepts, ## Architecture Decisions, or ## Constraints).
5. Construct the entry line using the exact format from existing entries:
   - Domain concepts: `- [{Title}]({relative-path}) — {one-line description} [public] [verified: {today}]`
   - Decisions: `- [{NNN}: {Title}]({relative-path}) — {one-line description} [verified: {today}]`
   - Constraints: `- [{Title}]({relative-path}) — {one-line description} [public] [verified: {today}]`
   Note: Use em dash ` — ` (not double dash) per research recommendation. Paths are relative to `.context/` (or `.context.local/`).
6. If the section contains `(none yet)` or a `(none` placeholder, replace that line with the new entry.
7. Otherwise, insert the new entry as the last line before the next `##` header (or end of file if last section).

**Step 12: Summary and commit prompt.**
1. Display a summary:
   ```
   Entry created:
     Type:     {Domain concept|Architecture decision|Constraint}
     File:     {file path}
     Manifest: {MANIFEST.md path} updated
   ```
2. Use AskUserQuestion: "Commit the new entry?"
   - Options: "Yes (recommended)", "No"
   - If yes: stage the created file and the updated MANIFEST.md, commit with message `docs: add {type} — {title}`

Important implementation notes:
- Follow the same prose style as dc:init and dc:validate -- numbered steps, clear conditionals, explicit tool usage.
- The skill file should be approximately 150-200 lines (similar in scope to dc:validate at 277 lines, but dc:add has less conditional fix flow complexity).
- Do NOT include code blocks with actual implementation logic -- the skill file contains instructions that Claude follows, not executable code.
- Use the same formatting patterns: `## Step N:` headers, numbered substeps, conditional branches with "If... then..." language.
