Create `commands/dc/explore.md` following the exact structure of `commands/dc/init.md` (YAML frontmatter + objective + execution_context + process sections).

**Frontmatter:**
```yaml
name: dc:explore
description: Browse and search domain context in the current project. Shows summary of all entries with freshness status, supports keyword search, and lets you drill into specific entries. Use when you want to understand what domain knowledge exists.
allowed-tools:
  - Read
  - Glob
  - Bash
  - AskUserQuestion
```

Note: No Write tool -- dc:explore is read-only.

**Objective section:** Browse and search domain context. Show summary with freshness, support keyword search, progressive disclosure for drilling into entries.

**Execution context section:** Explain that .context/MANIFEST.md is the primary data source. Entry line formats (linked entries, module context entries, empty sections). Paths in MANIFEST.md are relative to .context/ directory.

**Process section -- implement these steps:**

**Step 1: Check for .context/ directory**
- If `.context/` does not exist: display "No .context/ directory found. Run /dc:init to set up domain context." and stop.
- This handles EXPL-04.

**Step 2: Check for keyword argument**
- If the user provided a keyword argument (e.g., `/dc:explore integration`), go to Step 6 (keyword search).
- If no keyword, continue to Step 3 (summary).

**Step 3: Read and parse MANIFEST.md**
- Read `.context/MANIFEST.md`.
- Parse entries under each section header: `## Domain Concepts`, `## Architecture Decisions`, `## Constraints`, `## Module Context Files`.
- For each entry line starting with `- [` (linked) or `- ` followed by a path (module context):
  - Extract: name, path, verified date.
  - Lines like `(none yet)` or `(none...)` are NOT entries -- they indicate empty sections.
- For each entry, compute freshness:
  - Parse `[verified: YYYY-MM-DD]`.
  - Calculate days between verified date and today.
  - If >90 days: mark as stale with day count.
  - If no verified date found: mark as `[no date]`.

**Step 4: Display summary**
- Format output exactly as specified (from CONTEXT.md decisions):

```
Domain Context:

  Domain Concepts (N)
    - Entry Name                    [verified: YYYY-MM-DD]
    - Another Entry                 [STALE - 94 days]

  Architecture Decisions (N)
    - 001: Title                    [verified: YYYY-MM-DD]

  Constraints (N)

  Module Context Files (N)
    - path/to/CONTEXT.md            [verified: YYYY-MM-DD]
```

- Each section shows count in parentheses.
- Empty sections show count (0) with no entries listed.
- Entry names left-aligned, freshness right-aligned with consistent spacing.
- No descriptions in the summary -- names and freshness only.
- This handles EXPL-01, EXPL-02, EXPL-06.

**Step 5: Progressive disclosure (drill-in)**
- After showing the summary, use AskUserQuestion to offer drill-in.
- Prompt: "Explore an entry?"
- First level -- section selection (respecting ~4 option limit):
  - Only show sections that have entries (count > 0), plus "Done".
  - If 4+ non-empty sections: group last section with Done or use pagination.
  - Recommended option: the first non-empty section.
- If user picks "Done": stop.
- Second level -- entry selection within chosen section:
  - If section has 1-2 entries: list entries + "Back" + "Done" (4 options max).
  - If section has 3 entries: list entries + "Back" (4 options, no separate Done since Back returns to section level).
  - If section has 4+ entries: show first 2 entries + "More" + "Back". "More" shows next 2, etc.
  - Recommended option: first entry.
- When user picks an entry:
  - Read the full file content (resolve path relative to `.context/` directory).
  - Display the entry's full content. Show the file path as a header.
  - After displaying, loop back: "Explore another entry?" and return to section selection.
- This handles EXPL-05.

**Step 6: Keyword search**
- Read `.context/MANIFEST.md` and parse all entries (same as Step 3).
- For each entry, check if the keyword matches (case-insensitive substring):
  1. Entry name
  2. Entry description (from MANIFEST.md line, after the em dash)
  3. File content (read each entry's file and search)
- Collect all matches with match location (name/description/content).
- If no matches: display "No entries matching '[keyword]' found. Run /dc:explore to see all entries." and stop.
- If exactly one match: read and display the full file content immediately.
- If multiple matches: list matching entries with match location, then use AskUserQuestion to let user pick which to view. After viewing, offer to view another match or stop.
- No fuzzy matching or "did you mean" suggestions.
- This handles EXPL-03.

**Important implementation notes:**
- Keep the skill file concise and intent-based -- Claude is the runtime, not a parser.
- Follow dc:init's prose style: numbered steps, clear conditionals, specific output formats.
- The skill file should be roughly 120-180 lines (comparable to dc:init at ~187 lines).
- Do NOT include Write in allowed-tools -- dc:explore is read-only.
