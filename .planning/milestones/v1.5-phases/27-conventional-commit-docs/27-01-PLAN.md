  <action>
Create CONTRIBUTING.md at the repo root with the following content:

1. **Header** — "Contributing" title, brief intro saying this project uses conventional commits for automated versioning.

2. **Commit Message Format** section — explain the format: `type(scope): description` where scope is optional. Show that the first line is the subject, then optional body, then optional footer.

3. **Commit Types** section — table of all 10 recognized types from release-please-config.json:
   - Visible in changelog: feat (Features), fix (Bug Fixes), perf (Performance), docs (Documentation)
   - Hidden from changelog: chore, ci, test, refactor, style, build
   - Explain "hidden" means these still follow conventional format but don't appear in release notes.

4. **Version Bumps** section — explain the mapping:
   - `feat:` commits trigger a MINOR version bump (1.x.0)
   - `fix:`, `perf:`, `docs:` and other types trigger a PATCH version bump (1.0.x)
   - `BREAKING CHANGE:` footer or `!` after type (e.g., `feat!:`) triggers a MAJOR version bump (x.0.0)

5. **Examples** section — 3-4 concrete commit message examples:
   - A simple feature: `feat: add dc:validate skill`
   - A fix with scope: `fix(hooks): handle missing stdin gracefully`
   - A breaking change: `feat!: rename dc:init to dc:setup` with BREAKING CHANGE footer
   - A hidden type: `chore: update dev dependencies`

Keep it concise and practical. No need for development setup instructions or other contributing guidelines beyond commit format — this project's build/run info is in AGENTS.md.
  </action>
