Create `commands/dc/extract.md` following the Claude Code skill format (YAML frontmatter + objective/execution_context/process sections).

**Frontmatter:**
- name: dc:extract
- description: Extract domain knowledge from completed GSD phases into .context/ entries. Scans planning artifacts, classifies findings, and lets you selectively create domain concepts, decisions, and constraints.
- allowed-tools: Read, Write, Edit, Bash, Glob, AskUserQuestion

**Objective section:** Extract durable domain knowledge from completed GSD phase artifacts (.planning/phases/) into permanent .context/ entries. Scans CONTEXT.md, SUMMARY.md, RESEARCH.md, and RETROSPECTIVE.md from completed phases, classifies findings into domain concepts, decisions, and constraints, previews proposals with source attribution, and creates spec-compliant files for accepted proposals.

**Execution context section:** Include:
- Template resolution pattern (same as dc:add Step 2 — local then global)
- MANIFEST.md section headers and entry line formats (copied from dc:add's execution_context)
- Path-to-section mapping table (domain/ -> Domain Concepts, decisions/ -> Architecture Decisions, constraints/ -> Constraints)
- Extraction criteria definitions:
  - Domain concepts: business rules, invariants, lifecycle models, domain terminology. NOT code patterns or implementation details.
  - Architecture decisions: significant tradeoffs with alternatives considered, lasting impact on system direction. NOT implementation choices like variable naming.
  - Constraints: external requirements from regulatory, API limits, security policies, third-party contracts. NOT internal conventions.
- Phase artifact locations: `.planning/phases/{N}-{name}/` with file patterns `{N}-CONTEXT.md`, `{N}-*-SUMMARY.md`, `{N}-RESEARCH.md`, `{N}-RETROSPECTIVE.md`

**Process section — implement these steps:**

**Step 1: Check Prerequisites (SCAN-02)**
Check `.context/` directory exists. If not: "No .context/ directory found. Run /dc:init to set up domain context first." and stop.
Check `.planning/phases/` directory exists. If not: "No .planning/phases/ directory found. This skill extracts knowledge from GSD planning artifacts. Set up GSD first." and stop.

**Step 2: Parse Phase Scope (SCAN-03)**
Check if a range argument was provided:
- No argument: scan all completed phases
- Single number (e.g., `7`): scan only that phase
- Range (e.g., `7-9`): scan phases 7 through 9
Extract phase number from directory name prefix (e.g., `14-gsd-bridge-template` -> 14). Use Glob to find `.planning/phases/*/` directories.

**Step 3: Identify Completed Phases (SCAN-01)**
For each phase directory (filtered by scope from Step 2), check for presence of any `*-SUMMARY.md` file. Only phases with at least one SUMMARY.md are considered completed. If no completed phases found in scope: "No completed phase artifacts found in .planning/" and stop.

**Step 4: Read Phase Artifacts (SCAN-01)**
For each completed phase, read available artifacts:
- `{N}-CONTEXT.md` — focus on `<decisions>` and `<domain>` blocks
- `{N}-*-SUMMARY.md` — focus on `key-decisions` and `patterns-established` in frontmatter, and `## Decisions Made` section
- `{N}-RESEARCH.md` — focus on architecture patterns and pitfalls (if present)
- `{N}-RETROSPECTIVE.md` — focus on lessons learned and patterns (if present)

**Step 5: Resolve Template Path**
Same as dc:add Step 2: check local `.claude/domain-context/templates/` then global `~/.claude/domain-context/templates/`. If neither exists, show installer message and stop.

**Step 6: Classify Extractable Knowledge (CLASS-01, CLASS-02, CLASS-03)**
Analyze all collected artifact content. Identify extractable entries using the criteria from the execution_context section. For each finding, record: type (concept/decision/constraint), proposed title, one-line description, source (phase number + file), and key content points.

Important: extract WHY, not WHAT or HOW. Filter out implementation details, code patterns, file structures, and task lists. Focus on business rules, decision rationale, and external constraints.

**Step 7: Cross-Reference MANIFEST.md (PROP-01)**
Read `.context/MANIFEST.md`. Parse entries under each section header (same parsing as dc:explore). For each proposal, check if any existing entry name matches the proposal title (case-insensitive substring match). If a match is found, mark the proposal as "skipped — already in MANIFEST.md" and exclude it from the proposal list.

**Step 8: Present Proposal Table (PROP-02)**
Display all proposals in a summary table:
```
Proposed extractions from phases {range}:

  #  Type        Title                    Source
  1  Decision    {title}                  Phase {N} {file}
  2  Concept     {title}                  Phase {N} {file}
  ...

  {N} proposals ({breakdown by type})
  Skipped: "{title}" (already in MANIFEST.md)
```
If no proposals remain after filtering: "No new domain knowledge found to extract from the scanned phases." and stop.

**Step 9: Accept/Reject Loop (PROP-03)**
For each proposal, show a detailed preview:
- Type, title, source phase and file
- Content preview showing the key content that would be filled into the template sections
- Use AskUserQuestion with options: "Accept (recommended)", "Reject", "Edit first"
- If "Edit first": collect freeform corrections via AskUserQuestion, apply changes, show updated preview with "Accept" / "Cancel" options
- Track which proposals were accepted

If no proposals accepted: "No proposals accepted. Nothing to extract." and stop.

**Step 10: Write Accepted Entries (PROP-04)**
Write all accepted proposals AFTER the full accept/reject loop completes (not incrementally). For each accepted proposal:

1. Read the appropriate template from TEMPLATE_DIR (domain-concept.md, decision.md, or constraint.md)
2. Fill template sections with extracted content
3. Replace `{verified_date}` with today's date (YYYY-MM-DD)
4. Strip HTML comments except `<!-- verified: ... -->` lines
5. Derive kebab-case filename from title (same as dc:add Step 5)
6. For decisions: auto-detect next ADR number (same as dc:add Step 6 — Glob `.context/decisions/[0-9][0-9][0-9]-*.md`, max+1, zero-pad 3 digits)
7. Write file to `.context/{subdir}/{filename}.md`

**Step 11: Update MANIFEST.md (PROP-04)**
For each written file, update `.context/MANIFEST.md`:
1. Find the correct section header
2. Construct entry line using the exact format from dc:add (with em dash separator)
3. If section contains `(none yet)`, replace that line
4. Otherwise append before next `##` header

**Step 12: Post-Extraction Summary (PROP-05)**
Display summary:
```
Extraction complete:
  {N} domain concept(s) created
  {M} architecture decision(s) created (ADR-{numbers})
  {K} constraint(s) created

  Files:
    {list of created file paths}

  MANIFEST.md updated with {total} new entries
```

**Step 13: Commit Prompt**
Use AskUserQuestion: "Commit extracted entries?" with options "Yes (recommended)", "No".
If "Yes": stage all created files and MANIFEST.md with `git add`, commit with message `docs: extract {N} domain context entries from phases {range}`.

**Style notes:**
- Follow the same writing style as dc:add.md — imperative instructions, numbered sub-steps, specific format strings
- Each step should be a `## Step N:` heading inside `<process>`
- Keep step descriptions precise enough that Claude can execute without ambiguity
- Reference the execution_context section for format strings rather than repeating them

The validate-templates.sh script checks for existence of skill/template files but does NOT currently check command files in `commands/dc/`. However, it does verify template files in `templates/`.

Since dc:extract is a skill file (not a template), the validation approach should be lightweight. Add a new section to validate-templates.sh after the existing checks (before the Summary section):

```bash
# ── 6. Skill file existence ──────────────────────────────────────────────
echo -e "\n${BOLD}=== Skill Files ===${RESET}\n"

SKILL_FILES=(add.md explore.md init.md refresh.md validate.md extract.md)
SKILLS_DIR="${SCRIPT_DIR}/../commands/dc"

for f in "${SKILL_FILES[@]}"; do
  if [ -f "${SKILLS_DIR}/${f}" ]; then
    pass "${f} exists"
  else
    fail "${f} missing"
  fi
done
```

This validates that all dc:* skill files exist, including the new extract.md. Also check each skill file has required frontmatter fields (name, description, allowed-tools) and required XML sections (objective, execution_context, process):

```bash
# Check skill frontmatter and sections
for f in "${SKILL_FILES[@]}"; do
  filepath="${SKILLS_DIR}/${f}"
  if [ -f "$filepath" ]; then
    if grep -q "^name:" "$filepath"; then
      pass "${f} has name: frontmatter"
    else
      fail "${f} missing name: frontmatter"
    fi
    if grep -q "<process>" "$filepath"; then
      pass "${f} has process section"
    else
      fail "${f} missing process section"
    fi
  fi
done
```
