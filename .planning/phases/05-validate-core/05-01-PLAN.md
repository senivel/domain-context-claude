<task type="auto">
  <name>Task 1: Create dc:validate skill file</name>
  <files>commands/dc/validate.md</files>
  <action>
Create `commands/dc/validate.md` following the established skill file pattern (YAML frontmatter + objective + execution_context + process sections).

**Frontmatter:**
- name: dc:validate
- description: Check structural integrity of domain context. Reports broken links in MANIFEST.md, orphan files not referenced in the manifest, and stale entries.
- allowed-tools: Read, Glob (only these two -- no Write, no Bash, no AskUserQuestion; Phase 5 is read-only)

**Objective:** Check structural integrity of domain context and report issues grouped by check type with severity levels.

**Execution context:** Reuse the same MANIFEST.md entry format documentation from dc:explore (linked entries, decisions, module context entries, empty section handling). Include explicit path resolution notes:
- MANIFEST.md paths are relative to `.context/`. Prepend `.context/` when checking file existence.
- Module Context File paths may use `../` to reference files outside `.context/`. Resolve relative to `.context/` directory to get project-root-relative paths.
- Glob results are project-root-relative.

**Process steps:**

Step 1: Check for .context/ directory. If absent: "No .context/ directory found. Run /dc:init to set up domain context." Then stop.

Step 2: Read and parse MANIFEST.md. Parse entries under the four section headers (Domain Concepts, Architecture Decisions, Constraints, Module Context Files). For each entry extract: name, file path, verified date. Skip `(none yet)` placeholders. Same parsing as dc:explore Step 3.

Step 3: Broken Links check (VALD-01). For each parsed entry, resolve the path relative to `.context/` and attempt to read the file. If the file does not exist, record as a broken link with the entry name and path.

Step 4: Orphan Files check (VALD-02).
- Glob `.context/domain/*.md`, `.context/decisions/*.md`, `.context/constraints/*.md`
- Remove .gitkeep files from results
- For each file found, convert to path relative to `.context/` (e.g., `domain/integration-model.md`) and check if any MANIFEST.md entry references this path. If not referenced: orphan.
- Also discover CONTEXT.md files using Glob `**/CONTEXT.md` with exclusions for `.context/`, `node_modules/`, `.git/`, `.planning/` (same as dc:explore Step 3.5). For each discovered CONTEXT.md, check if any Module Context Files entry resolves to this path. If not referenced: orphan. Show these as project-root-relative paths (e.g., `src/auth/CONTEXT.md`).

Step 5: Stale Entries check (VALD-03). For each parsed entry with a verified date, compute days since verification. If >90 days: stale. Entries with no verified date: treat as a warning ("no verified date"). Compute total entries validated (entries that have a parseable verified date or path, excluding empty-section placeholders).

Step 6: Format and display results. Group by check type per locked decisions in CONTEXT.md:

```
Domain Context Validation:

  {icon} Broken Links ({count})
    - {Entry Name} -> {path}

  {icon} Orphan Files ({count})
    - {file path}

  {icon} Stale Entries ({count})
    - {Entry Name} [verified: {date}, {N} days ago]

  {summary line}
```

Icons: checkmark for pass (0 issues), X for errors (broken links, orphans), warning symbol for warnings (stale entries).
Summary: If issues found, show "{N} errors, {M} warnings" using severity. If clean: "All checks passed. {N} entries validated."
Show all three groups even when counts are 0 (clean state shows all three with checkmark).
For groups with 0 items, show just the header line with (0), no items beneath.

**Important anti-patterns to call out in the skill file:**
- Do NOT check the Access Levels section (metadata, not entries)
- Do NOT offer to fix issues (that is Phase 6 / VALD-05)
- Do NOT flag .gitkeep files as orphans
- Do NOT flag files in .context/ root (like MANIFEST.md itself) as orphans -- only scan subdirectories
