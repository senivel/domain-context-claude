  <action>
Create `rules/dc-context-editing.md` with the following structure:

1. YAML frontmatter with `globs: .context/**, **/CONTEXT.md` (comma-separated, unquoted patterns). Use `globs:` NOT `paths:` -- the `paths:` key has parser bugs (GitHub issue #17204). Do NOT quote the glob patterns -- the CSV parser strips quotes incorrectly.

2. An H1 title: `# Domain Context Formatting`

3. A one-line reference to the authoritative spec: `Authoritative spec: ~/code/domain-context/SPEC.md`

4. Concise bullet-point guidance organized by file type (per research recommendation), covering these sections:
   - **General**: kebab-case filenames, verified date format (`<!-- verified: YYYY-MM-DD -->` as HTML comment after H1), MANIFEST.md entry verified format (`[verified: YYYY-MM-DD]` inline), token budget (~500-1000 tokens per file, max 1500)
   - **MANIFEST.md**: one-line blockquote description after H1, required sections (Access Levels, Domain Concepts, Architecture Decisions, Constraints, Module Context Files), entry format with path, description, access level, verified date, ADR entry format with NNN prefix, always update MANIFEST.md when adding/modifying context files
   - **Domain Concepts** (.context/domain/): required sections (What This Is, Key Attributes, Business Rules, Invariants, Related Concepts), optional Lifecycle section, business rules as numbered testable statements, one concept per file, split if >2000 tokens
   - **Decisions** (.context/decisions/): filename format `{NNN}-{slug}.md` (zero-padded three digits), title format `ADR-{NNN}: {Title}`, required sections (Status, Context, Decision, Rationale, Consequences, Affected Modules), status values (accepted | superseded | deprecated)
   - **Constraints** (.context/constraints/): required sections (Source, Requirements, Impact on Code, Verification)
   - **Module CONTEXT.md**: required sections (What This Module Does, Domain Concepts Owned, Business Rules, Non-Obvious Decisions, What This Module Does NOT Do, Dependencies)

5. Target ~30-40 lines of content (not counting frontmatter). Brevity is critical because this rule loads into context on EVERY matching file read. Use terse bullets, not explanatory paragraphs. Reference the spec for edge cases rather than duplicating it.

Read the template files (@templates/) to verify the exact section names and formats before writing the rule. The rule content must accurately reflect the spec templates.
