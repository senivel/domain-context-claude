  <action>
Create the `commands/dc/` directory (first skill directory in this project), then create `commands/dc/init.md`.

The skill file must follow Claude Code skill format with these sections:

**YAML Frontmatter:**
```yaml
name: dc:init
description: Initialize Domain Context in the current project. Creates .context/ directory with MANIFEST.md, domain/, decisions/, and constraints/ subdirectories. Scaffolds ARCHITECTURE.md, wires AGENTS.md with domain-context snippet, and creates CLAUDE.md with @AGENTS.md pointer. This skill should be used when setting up domain context on a new or existing project.
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
  - AskUserQuestion
```

**`<objective>` section:** Initialize a complete Domain Context setup in the current working directory. Create .context/ structure, fill templates with inferred project metadata, and wire root-level files (ARCHITECTURE.md, AGENTS.md, CLAUDE.md, .gitignore).

**`<execution_context>` section:** Reference template locations — local (.claude/domain-context/templates/) and global (~/.claude/domain-context/templates/) install paths. Note that templates use {placeholder} tokens and HTML guidance comments that must be stripped when filling.

**`<process>` section — must include these steps in order:**

**Step 1: Resolve Template Path**
- Check .claude/domain-context/templates/ (local) first
- Fall back to ~/.claude/domain-context/templates/ (global)
- If neither exists, tell user to run the installer and stop

**Step 2: Basic Safety Check**
- If .context/MANIFEST.md already exists, warn the user using AskUserQuestion: "This project already has a .context/ directory. Proceed anyway? (Recommended: No — use /dc:explore to review existing context)" with options: Yes (proceed), No (stop). This is a minimal guard before Phase 3 adds full idempotency.

**Step 3: Infer Project Metadata**
- Try sources in order for project description (one_line_description):
  1. package.json "description" field
  2. pyproject.toml [project].description
  3. Cargo.toml [package].description
  4. composer.json "description" field
  5. setup.cfg [metadata].description
  6. go.mod module path (derive project name)
  7. README.md first substantive paragraph (skip badges, headings, blank lines)
  8. Directory name, converted to title case with hyphens/underscores replaced by spaces
- If none produce useful description, use AskUserQuestion with GSD interaction pattern: concrete prompt "I couldn't detect a project description automatically. What does this project do?", recommended option to type freeform description, built-in "Other" escape
- For {system_purpose}: expand the same description to 2-4 sentences
- For {restricted_context_instructions}: use "Contact the project maintainer for access to restricted context."
- For remaining ARCHITECTURE.md placeholders ({module_rows}, {data_flow}, {key_boundaries}, {technology_decisions}): replace with TODO comments like "<!-- TODO: Document module map -->"

**Step 4: Create .context/ Directory Structure**
- Run `mkdir -p .context/domain .context/decisions .context/constraints`
- Create .context/domain/.gitkeep, .context/decisions/.gitkeep, .context/constraints/.gitkeep (empty files)
- Narrate: "Created .context/ directory with domain/, decisions/, constraints/ subdirectories"

**Step 5: Fill and Write MANIFEST.md**
- Read templates/manifest.md (from resolved TEMPLATE_DIR)
- Replace {one_line_description} with inferred description
- Replace {restricted_context_instructions} with the default text
- Strip all HTML comments (<!-- ... -->)
- Write to .context/MANIFEST.md
- Narrate: "Created .context/MANIFEST.md"

**Step 6: Fill and Write ARCHITECTURE.md** (only if file does not exist)
- Read templates/architecture.md from TEMPLATE_DIR
- Replace {system_purpose} with expanded description
- Replace {module_rows}, {data_flow}, {key_boundaries}, {technology_decisions} with TODO comments
- Strip all HTML comments
- Write to ARCHITECTURE.md
- Narrate: "Created ARCHITECTURE.md" or "ARCHITECTURE.md already exists — skipped"

**Step 7: Inject AGENTS.md Snippet**
- Read templates/agents-snippet.md from TEMPLATE_DIR
- Try to read AGENTS.md from project root
- If AGENTS.md does not exist: create it with just the snippet content. Narrate: "Created AGENTS.md with domain-context snippet"
- If AGENTS.md exists and contains "<!-- domain-context:start -->": narrate "AGENTS.md already has domain-context snippet — skipped"
- If AGENTS.md exists but no sentinel: read existing content, write existing content + two blank lines + snippet. Narrate: "Appended domain-context snippet to AGENTS.md"

**Step 8: Handle CLAUDE.md**
- Read templates/claude.md from TEMPLATE_DIR (contains "@AGENTS.md")
- Try to read CLAUDE.md from project root
- If CLAUDE.md does not exist: create with template content. Narrate: "Created CLAUDE.md"
- If CLAUDE.md exists and contains "@AGENTS.md": narrate "CLAUDE.md already references @AGENTS.md — skipped"
- If CLAUDE.md exists but no @AGENTS.md reference: append a blank line + "@AGENTS.md" to existing content. Narrate: "Added @AGENTS.md reference to CLAUDE.md"

**Step 9: Handle .gitignore**
- Try to read .gitignore from project root
- If .gitignore does not exist: create with ".context.local/" as content. Narrate: "Created .gitignore with .context.local/ entry"
- If .gitignore exists and contains ".context.local/": narrate ".gitignore already has .context.local/ — skipped"
- If .gitignore exists but no .context.local/ entry: append a blank line + ".context.local/" to existing content. Narrate: "Added .context.local/ to .gitignore"

**Step 10: Offer to Commit**
- Use AskUserQuestion with GSD interaction pattern: "Domain Context initialized successfully. Commit the created files?"
  - Option 1: "Yes (recommended)" — run `git add` on all created/modified files and commit with message "chore: initialize domain context"
  - Option 2: "No" — skip commit, files remain unstaged

**Writing style:** Use imperative/infinitive form throughout (e.g., "Read the template file", "Check if AGENTS.md exists", not "You should read" or "Claude reads"). This follows skill-creator best practices.

**Important constraints:**
- The skill must READ templates from the install location at runtime, never embed template content inline
- For AGENTS.md and CLAUDE.md, only append or create — never rewrite or reorder existing content
- All narration is inline during the process (not a summary at the end — that's Phase 3)
- Keep the skill under 300 lines — the logic is procedural, not branching
